# Decision 5 — Tuning loop: baseline, epsilon acceptance, rails, finalize

**Decision.** Package `internal/loop` owns the cycle. All arithmetic is
plain and deterministic; no judgment calls.

**Baseline.** Evaluate the incumbent (`prompt.txt`) over dev `--repeat` times
(default 3). Each evaluation yields a composite (mean case score).
`baseline = mean(composites)`; `epsilon = max(composites) - min(composites)`
(the run-to-run noise floor). `best` starts at `baseline` with the incumbent
as the best prompt.

**Iteration.** Forever: check rails, then one improver call (D06) yields a
candidate; evaluate the candidate over dev once; **accept iff
`candidate > best + epsilon`** (strict). On accept the candidate becomes the
best prompt and `best` its composite. Rejects change nothing. There is no
reject-streak auto-stop (prior art removed it).

**Graceful degradation on an unparseable improver reply.** A malformed
improver response is not fatal. When `Propose` returns an error wrapping
`ErrUnparseable` (D06) and the context is not cancelled, the loop **skips that
iteration and continues**: it records a skip line in `history.md`, increments
`Outcome.Skipped`, and writes the parse cause to `Err`. The skipped iteration
still counts — it consumed an iteration and its improver spend — so a
persistently-failing improver terminates the run through the ordinary rails
with the best-so-far winner finalized, rather than aborting a good run. A skip
is **not** added to the attempt history shown to the improver (it has no
summary or composite). The one exception, to keep faith with *fail loudly*:
`maxConsecutiveSkips` (3) skips **in a row** — an improver that never produces
a parseable reply — is genuinely unrecoverable and stops with exit 1 (stop
reason `improver unrecoverable`), so an unbounded run cannot burn its whole
budget on replies it can never parse. Any successful proposal resets the
consecutive count. A hard improver call or context error is unchanged: it still
stops the run (internal failure / interrupted).

**Rails and stops.** Checked at the top of every iteration, fixed order for
determinism: `--max-iterations`, `--max-time`, `--max-spend` (runner +
improver costs combined, accumulated from each tracked conversation's
`TotalCost()`); all default 0 = unlimited. Spend figures are real because
every conversation carries catalog pricing (D02) — agentkit computes each
turn's cost from token usage; provider self-report exists only on
openrouter — and `--max-spend` refuses to arm at all when a configured
model is unpriceable (D02's `PricingPrecheck`, run before the loop starts).
Also stops: `best == 1.0` (perfect dev score, exit 0) and context
cancellation from SIGINT/SIGTERM (exit 130). Rail crossing exits 3.

**Finalize — always runs**, on every stop reason including interrupt and
rails (structured as a single exit path, ralph's `finish` pattern):

- If at least one candidate was accepted and holdout is non-empty: evaluate
  the winner on holdout **exactly once** — the only time holdout is ever
  touched; the improver never sees it. Verdict: `holdout <= baseline` ⇒
  `OVERFIT`, else `generalized`.
- Write `summary.md`, print the final report and the unified diff incumbent
  → winner (D07/D08). With no accepted candidate: report "no improvement
  found" honestly, skip holdout, exit 0 (that is a valid answer, not a
  failure).

The final report **always terminates with a `stop: <reason>` line** on every
path — accepted-winner and no-improvement alike — and the winner diff is
printed *before* that line, never after. (Prior art printed the diff last on
the accepted path with no trailing stop line, so a run that ended with a winner
looked truncated mid-diff.) When any iteration was skipped, the report and
`summary.md` carry a `skipped: <n>` count.

**Warnings surfacing.** agentkit's provider warnings (`Stream.Warnings()`
— setting degradation, unknown cost) must reach the operator; dropping
them is what hid the $0-cost defect this design corrects. The runner and
improver report every warning through the `runner.WarnFunc` seam (D03,
D06); `Run` wires both to one per-run sink that writes
`warning: <setting> — <detail>` to `Deps.Err`, **deduplicated by warning
code**: the first occurrence of a code per run is written, repeats are
dropped, distinct codes each get their own line. All codes surface; none
are filtered.

**Failure reporting.** Failing loudly means the *cause* reaches the
operator, not only the classification. The loop's `Deps` carries an
`Err io.Writer` (the composition root wires stderr; `io.Discard` when nil).
Every failure stop — evaluation error, improver error, workspace write
error, holdout-evaluation error — writes the underlying error text to
`Err` before the final report. A stop line like `stop: internal failure`
must never be the only trace of what went wrong.

```go
package loop

type Rails struct{ MaxIterations int; MaxTime time.Duration; MaxSpend float64 }
type Deps struct {
    // ...conversation seams, scorer, workspace, clock...
    Out io.Writer // health log
    Err io.Writer // failure causes; stderr in the real binary
}
type Outcome struct {
    Baseline, Epsilon, Best float64
    Accepted  int
    Holdout   *float64 // nil if not run
    Verdict   string   // "", "OVERFIT", "generalized"
    StopReason string  // maps to exit code
}
func Run(ctx context.Context, deps Deps, f *folder.Folder, cfg *config.Config, opts Options) (Outcome, int)
```

**Rejected.** Accept on `>=` or on raw mean without epsilon (prior art's
real run shows an inside-epsilon candidate correctly rejected). Holdout per
iteration (turns holdout into a second dev set). Reject-streak auto-stop
(tried and removed in prior art). Resume of interrupted runs (v1 re-baselines
every invocation, which also re-measures epsilon honestly). Per-turn warning
lines (a degraded setting repeats every call — dedup by code keeps stderr
readable). Surfacing only cost warnings (the next silently-dropped code
would reopen this class of gap). Parsing warnings back out of
`Conversation.Log` JSONL (a typed callback at the drain site is direct).

  for improver, candidate-evaluation, workspace-write, and
  holdout-evaluation failures; through the composition root this reaches
  the process's stderr.
- R-SKPC-4XN1 — an improver reply that never parses (`Propose` returns

  writes its cause to `Err`, and the loop continues and ends via a normal

- R-SKPU-8QR2 — with no rails set, `maxConsecutiveSkips` consecutive

- R-FNST-6LM3 — the final report ends with a `stop: <reason>` line on both
